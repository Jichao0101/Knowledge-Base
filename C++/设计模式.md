# 1 面向对象设计原则
## 1.1 依赖倒置原则
- 高层稳定模块不应该依赖低层变化模块，二者都应该依赖于抽象
- 抽象（稳定）不应该依赖于实现细节（变化），实现细节应该依赖于抽象
## 1.2 开放封闭原则
- 对扩展开放，对更改封闭
- 类模块应该是可扩展的，但是不可修改
## 1.3 单一职责原则
- 一个类应该只有一个引起它变化的原因
- 变化的方向隐含着类的责任
## 1.4 Liskov替换原则
- 子类必须能够替换它们的基类（Is-A)
- 继承表达类型抽象
## 1.5 接口隔离原则
- 不应该强迫用户程序依赖它们不用的方法
- 接口应该小而完备
## 1.6 优先使用对象组合
- 类继承通常为白箱复用，对象组合通常为黑箱复用
- 继承在某种程度上破坏了封装性，子类父类耦合性高
- 对象组合只要求被组合的对象有良好定义的接口，耦合度低
## 1.7 封装变化点
- 使用封装来创建对象之间的分界层，让设计者可以在分界层的一侧进行修改，而不会对另一侧产生不良的影响，从而实现层次间的松耦合
- 即把**容易变化的决策/实现**隔离在**稳定的接口**之后
## 1.8 针对接口编程
- 不将变量类型声明为某个特定的具体类，而是声明为某个接口
- 客户程序无需获知对象的具体类型，只需要知道对象所具有的接口
- 减少系统中各部分的依赖关系，从而实现高内聚、松耦合的类型设计方案

# 2 模板模式
- 在确定稳定操作结构的前提下，灵活应对各个子步骤的变化或晚期实现需求
- ”不要调用我，让我来调用你“的反向控制结构是模板模式的典型应用
```
class BaseFatiguePipeline {
public:
  virtual ~BaseFatiguePipeline() = default;

  // 模板方法：固定流程骨架
  int32_t Process(std::shared_ptr<AtomicResult>& vision,
                  std::shared_ptr<VehicleState>& can,
                  std::shared_ptr<AlgorithmResult>& out) final
  {
    ComputeSignals(*vision, *can);        // 子类差异点 #1
    DecideFatigue(*vision, *can);         // 子类差异点 #2
    PostProcess(*vision, *can);           // 钩子：去抖、冷却、合并等级
    MakeResult(out);                      // 固化封装，避免子类乱改
    return 0;
  }

protected:
  // —— 可变步骤（抽象）——
  virtual void ComputeSignals(const AtomicResult& vision,
                              const VehicleState& can) = 0;
  virtual void DecideFatigue(const AtomicResult& vision,
                             const VehicleState& can) = 0;
  // —— 钩子（可覆盖，提供默认实现）——
  virtual void PostProcess(const AtomicResult&, const VehicleState&) {}
  // 结果构造固定在基类，保持输出一致性
  void MakeResult(std::shared_ptr<AlgorithmResult>& out) { }
```
