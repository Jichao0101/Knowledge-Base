完整的 C++ 测试流程：

- **gTest**：单元测试框架（写测试）
    
- **CTest**：CMake 的测试运行器（管测试）
    
- **覆盖率工具**（gcov / lcov / gcovr 等）：统计测试打中了多少代码
    

---

# 1 整体结构与测试流程

从高维视角看，测试流水线就是三步：

1. **构建（Build）**
    
    - CMake 配置工程（可选打开覆盖率）
        
    - 编译主库 + 测试可执行文件（基于 gTest）
        
2. **执行测试（Test）**
    
    - 本地调试：直接运行某个 gTest 可执行
        
    - 全量与 CI：通过 CTest 统一跑全部测试
        
3. **统计覆盖率（Coverage）**
    
    - 收集执行信息（`.gcda` / `.profraw` 等）
        
    - 用 lcov / gcovr / llvm-cov 生成报告
        
    - 结合 CI 设定覆盖率门槛（比如 <70% 不放行）

# 2 gTest：单元测试框架

## 2.1 gTest 

gTest 负责“**怎么写测试**”，提供：

- 断言：`EXPECT_EQ` / `ASSERT_TRUE` / `EXPECT_NEAR` …
    
- 测试用例：`TEST(SuiteName, CaseName)`
    
- 测试夹具：`TEST_F`，把公共初始化/清理封装在类里
    
- 失败信息打印
    
- 内建过滤运行：`--gtest_filter=Suite.Case`
    

## 2.2 基本示例

```
#include <gtest/gtest.h>

int add(int a, int b) {
    return a + b;
}

// 定义一个测试用例
TEST(MathTest, SimpleAdd) {
    EXPECT_EQ(add(1, 2), 3);
    EXPECT_EQ(add(-1, 1), 0);
}

// 初始化框架，并运行所有单元测试
int main(int argc, char **argv) {
    ::testing::InitGoogleTest(&argc, argv);
    return RUN_ALL_TESTS();
}
```

编译出 `test_math` 后：

```
./test_math                              # 运行所有测试
./test_math --gtest_filter=MathTest.*    # 只运行 MathTest 下的用例
./test_math --gtest_filter=MathTest.SimpleAdd
```

> 说明：如果链接了 `gtest_main`，可以省略 `main`，gTest 提供默认入口。

# 3 在 CMake 中集成 gTest

## 3.1 工程结构示例
```
project_root/
  src/
    math.cpp
  include/
    math.h
  test/
    test_math.cpp
  thirdparty/
    googletest/
      include/
      lib/
  CMakeLists.txt

```

## 3.2 CMake 配置示例

```
cmake_minimum_required(VERSION 3.10)
project(MyProject CXX)

set(CMAKE_CXX_STANDARD 14)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# 主库
add_library(mathlib src/math.cpp)
target_include_directories(mathlib PUBLIC ${PROJECT_SOURCE_DIR}/include)

# gTest 头文件 & 库路径
set(GTEST_INCLUDE_DIR ${PROJECT_SOURCE_DIR}/thirdparty/googletest/include)
set(GTEST_LIB_DIR     ${PROJECT_SOURCE_DIR}/thirdparty/googletest/lib)

include_directories(${GTEST_INCLUDE_DIR})
link_directories(${GTEST_LIB_DIR})

# 测试可执行文件
add_executable(test_math test/test_math.cpp)
target_link_libraries(test_math
    PRIVATE
        mathlib
        gtest
        gtest_main    # 若用了这个，可以删掉源码里的 main
        pthread
)

```

# 4 CTest：统一管理与调度测试

## 4.1 CTest 

- **CMake 自带的测试运行器**
    
- 不关心用 gTest、Catch2 还是自定义的 `assert`，它只负责：
    
    - 知道“有哪些测试可执行文件”
        
    - 按一定规则把它们跑一遍
        
    - 汇总结果
        

**一句话：gTest 负责“测试逻辑”，CTest 负责“怎么统一跑”。**

## 4.2 启用 CTest

在 CMake 中打开测试支持：

`enable_testing()`

## 4.3 注册 gTest 可执行为 CTest 测试

```
add_executable(test_math test/test_math.cpp)
target_link_libraries(test_math PRIVATE mathlib gtest gtest_main pthread)

add_test(NAME MathTests COMMAND test_math)
```

注册后，在 `build` 目录：

```
ctest            # 运行所有注册过的测试
ctest -R Math    # 名字匹配 Math 的测试
ctest -V         # 输出每个测试的详细 log
```

可以有多个测试二进制，比如：

```
add_executable(test_algorithm test/test_algorithm.cpp) target_link_libraries(test_algorithm PRIVATE algolib gtest gtest_main pthread) add_test(NAME AlgoTests COMMAND test_algorithm)
```

然后 `ctest` 会依次运行 `test_math`、`test_algorithm` 等一堆测试。

# 5 覆盖率（Coverage）：目的与流程

## 5.1 覆盖率

典型用途：

- 找到完全没被执行的函数 / 分支（风险点）
    
- 发现死代码 / 冗余逻辑
    
- CI 上做质量门槛：
    
    - 总行覆盖率 < 70% → CI fail
        
    - 核心模块 < 80% → 不允许合并
        

常见指标：

- **行覆盖率**：哪些行跑到了
    
- **分支覆盖率**：if / switch 的各个分支是否都有测试
    

## 5.2 在 CMake 中启用覆盖率编译

以 GCC 为例（Clang 类似）：

```
option(ENABLE_COVERAGE "Enable coverage build" OFF)
message(STATUS "ENABLE_COVERAGE = ${ENABLE_COVERAGE}")

if(ENABLE_COVERAGE)
    # 编译时插入计数逻辑
    add_compile_options(-fprofile-arcs -ftest-coverage)
    # 链接 gcov
    link_libraries(gcov)
endif()

```

构建时打开：

```
mkdir build_coverage
cd build_coverage
cmake .. -DCMAKE_BUILD_TYPE=Debug -DENABLE_COVERAGE=ON
make -j
```

此时：

- 每个编译单元会生成 `.gcno`（静态信息）
    
- 运行测试时会生成 `.gcda`（动态执行计数）

# 6 覆盖率报告生成（以 lcov 为例）

## 6.1 安装工具（Ubuntu 示例）

`sudo apt-get install lcov`

## 6.2 运行测试，产生数据

在 `build_coverage` 下：

`ctest          # 或者直接 ./test_math ./test_algorithm 等`

这一步执行覆盖率插桩的代码，生成 `.gcda` 数据文件。

## 6.3 收集覆盖率

`# 收集当前目录的覆盖率信息 lcov --capture --directory . --output-file coverage.info`

可选：过滤掉系统头文件 / 第三方库：

`lcov --remove coverage.info '/usr/*' '*/thirdparty/*' --output-file coverage_filtered.info`

## 6.4 生成 HTML 报告

`genhtml coverage_filtered.info --output-dir coverage_html`

然后浏览器打开：

`coverage_html/index.html`

就能看到“哪些行是绿色（被测到）”、“哪些是红色（没跑过）”。

# 7 gTest + CTest + 覆盖率：完整示意流程

可以按这样的节奏组织：

## 7.1 本地开发调试

```
mkdir build && cd build 
cmake .. -DCMAKE_BUILD_TYPE=Debug make -j  # 调试单个测试用例 
./test_math --gtest_filter=MathTest.SimpleAdd
```

## 7.2 本地全量测试（集成 CTest）

`ctest -V`

## 7.3 覆盖率构建 & 报告

```
mkdir build_coverage && cd build_coverage
cmake .. -DCMAKE_BUILD_TYPE=Debug -DENABLE_COVERAGE=ON
make -j
ctest                 # 跑所有测试

lcov --capture --directory . --output-file coverage.info
lcov --remove coverage.info '/usr/*' '*/thirdparty/*' --output-file coverage_filtered.info
genhtml coverage_filtered.info --output-dir coverage_html
```

# 8 gTest vs CTest

|维度|gTest|CTest|
|---|---|---|
|角色|单元测试框架（写测试、断言）|测试执行器（统一运行所有测试目标）|
|粒度|单个测试进程里的 TEST/TEST_F|多个测试可执行文件（test_math、test_algo 等）|
|运行方式|`./test_xxx --gtest_filter=...`|`ctest -R pattern`|
|输出风格|每条断言详细信息|测试集合结果汇总（pass/fail/时间）|
|与 CMake 关系|独立于 CMake，可单独使用|完全绑定 CMake，需要 `enable_testing`|
|在 CI 的地位|提供断言逻辑|提供统一入口与退出码，方便 CI 判断成功/失败|
