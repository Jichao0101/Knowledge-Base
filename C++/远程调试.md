
# 1 整体架构与角色分工

 - **板端（J6M）** 运行 ARM 架构可执行程序  
- **板端** 使用 `gdbserver` 托管被调试进程  
- **本地（x86 PC）** 使用同架构的 `gdb`（aarch64）  
- **VSCode** 作为调试前端，实现断点、单步、变量查看  

远程调试并非“本地调试远端程序”，而是 **调试职责的彻底解耦**：
```
┌────────────────────────┐
│ VSCode │
│ UI / 断点 / 源码视图 │
└──────────┬─────────────┘
│ GDB MI（Machine Interface）
┌──────────▼─────────────┐
│ 本地 GDB (aarch64) │
│ 符号解析 / DWARF / ABI │
└──────────┬─────────────┘
│ gdb remote protocol (TCP)
┌──────────▼─────────────┐
│ 板端 gdbserver │
│ ptrace / CPU / 内存 │
└──────────┬─────────────┘
│
┌──────────▼─────────────┐
│ ARM 可执行程序 (sdk) │
└────────────────────────┘
```

## 1.1 核心思想

> **“执行” 与 “理解” 分离**

- **板端**：只负责 *执行程序* 与 *执行调试指令*
- **本地**：只负责 *理解程序（符号、源码、变量）*
- **VSCode**：只是 gdb 的 UI 前端

---

# 2 工作原理（工程视角）

## 2.1 gdbserver 的职责（板端）

`gdbserver` 是一个 **极轻量级调试代理**：

- 通过 `ptrace` 控制目标进程
- 能力包括：
  - 暂停 / 继续执行
  - 读写寄存器
  - 读写进程内存
  - 插入/移除断点（指令级）
- **不解析 C/C++**
- **不需要符号信息**
- **不需要源码**

在 `gdbserver` 眼中，一切都是：
```
PC = 0xXXXXXXXX
REG[x0] = ...
MEM[0x...] = ...
```

---

## 2.2 本地 gdb 的职责（PC）

本地 gdb 才是真正的“调试大脑”：

- 读取 **ELF + DWARF 调试信息**
- 建立映射关系：
  - 源码行号 ↔ 指令地址
  - 变量名 ↔ 内存偏移
  - 函数 ↔ 符号
- 决定：
  - 断点位置
  - 单步语义
  - 栈回溯逻辑

**关键结论**：

> 本地必须持有 **同架构、同版本、带符号的可执行文件**

---

## 2.3 远程调试协议（关键机制）

两端通过 **GDB Remote Serial Protocol（RSP）** 通信：

- 本地 gdb 将调试意图翻译为协议命令
- gdbserver 执行并返回结果

示例（逻辑层面）：

1. VSCode 打断点：`main.cpp:123`
2. 本地 gdb：
   - 查 DWARF
   - 得到 ARM 指令地址 `0xABCDEF`
3. gdb → gdbserver：`Z0,ABCDEF,4`
4. 板端写入 trap 指令
5. CPU 命中断点
6. gdbserver 回传寄存器/内存状态
7. 本地 gdb 反解为源码与变量
8. VSCode 显示断点命中

# 3 架构与版本一致性要求

## 3.1 必须一致的部分

- **CPU 架构**  
- AArch64 ↔ AArch64
- **ABI / 调用约定**
- **寄存器模型**

## 3.2 不要求一致的部分

- Linux 发行版（Ubuntu / Yocto / BusyBox）
- 板端是否 strip
- 库路径（只影响调试动态库）

---

# 4 板端部署 gdbserver

## 4.1 方法1：直接启动调试

```bash
cd /userdata/jichao
gdbserver 0.0.0.0:2345 ./sdk <args>
```

## 4.2 方法2：附加到已有进程
```
pidof sdk
gdbserver 0.0.0.0:2345 --attach <PID>
```

# 5 本地 GDB 与 VSCode 连接

## 5.1 本地 gdb 选择

gdb-multiarch 或 aarch64-none-linux-gnu-gdb

## 5.2 VSCode 调试配置（launch.json）

```
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "J6M Remote Debug (aarch64)",
      "type": "cppdbg",
      "request": "launch",
      "program": "/home/user/open_explorer/build/sdk",
      "MIMode": "gdb",
      "miDebuggerPath": "/usr/bin/gdb-multiarch",
      "miDebuggerServerAddress": "192.168.55.62:2345",
      "cwd": "/home/user/open_explorer",
      "stopAtEntry": false,
      "setupCommands": [
        { "text": "set architecture aarch64" },
        { "text": "set substitute-path /open_explorer /home/user/open_explorer" },
        { "text": "-enable-pretty-printing" }
      ]
    }
  ]
}
```

# 6 关键工程要点（必须满足）

## 6.1 调试符号

- 编译必须带 -g
- 本地使用 未 strip 或带 debug symbol 的可执行文件
- 板端可运行 strip 版本

## 6.2 源码路径映射

编译路径 ≠ 本地路径时，必须设置：
`set substitute-path <build_path> <local_path>`
否则断点无法命中。

## 6.3 动态库调试（可选）

若需调 libc / libstdc++ / so：
把板端 rootfs（或 SDK/yocto 生成的 sysroot）放在本地某个目录/path/to/j6m_rootfs
```
set sysroot /path/to/j6m_rootfs
set solib-search-path /path/to/j6m_rootfs/lib:/path/to/j6m_rootfs/usr/lib
```

# 7 常见问题与原因

|现象|原因|
|---|---|
|断点灰色|本地无符号 / 路径不匹配|
|attach 失败|ptrace 限制|
|变量无法展开|无符号 / ABI 不一致|
|网络断开调试中断|TCP 连接断开|
