> **目标**  
> 确保模型在**训练态与部署态**之间保持可控的一致性；  
> 在 **pth → onnx → bc → hbm** 的每一个转换阶段，都能**量化评估精度变化、定位精度下降原因，并给出是否放行的工程依据**。

# 2 评估阶段划分

评估贯穿模型生命周期，共分为 **4 个主阶段**（Stage C 内部进一步细分）：

|阶段|模型形态|推理后端|输入形式|评估目的|
|---|---|---|---|---|
|Stage A|`.pth`|PyTorch|RGB（标准预处理）|建立精度基线|
|Stage B|`.onnx`|ONNX Runtime|RGB（同 Stage A）|验证导出一致性（应几乎不掉点）|
|Stage C|`.bc`|编译后模型|**YUV/NV12（前处理入图）**|验证“编译/融合/量化/前处理入图”的综合影响|
|Stage D|`.hbm`|最终 SDK|真实上板输入路径（YUV/NV12）|验证部署态精度（最终裁决）|

> ✅ Stage B 的意义：隔离变量，证明“导出 ONNX 本身没问题”。  
> ✅ Stage C 的意义：验证“工具链改图 + 编译 + 量化 + 前处理入图”引入的变化是否可控。

# 3 评估数据集设计

## 3.1 标准评估集（Primary Eval Set）

- 来自训练任务的 **val / test 集**
    
- 覆盖：
    
    - 常规场景
        
    - 各类别样本
        
    - 主流尺度分布
        

用于：

- mAP、Precision、Recall 等**主指标评估**
    

---

## 3.2 固定回归集（Regression Set）

> 强烈推荐（转化链路的“显微镜”）

- 50～200 张图片
    
- 固定不变，长期使用
    
- 覆盖：
    
    - 小目标
        
    - 遮挡
        
    - 低光 / 逆光
        
    - 高密度目标
        
    - 极端边界情况
        

用于：

- 转换版本回归测试
    
- 精度掉点定位（逐图对齐、误差统计、可视化）
    

---

## 3.3 校准集（Calibration Set，仅用于量化/校准）

- 通常取自训练集或与训练分布高度一致的数据
    
- 规模建议：200～2000 张（按模型大小与量化策略调整）
    
- **注意**：校准集不是用来“汇报精度”的，是用来让量化参数逼近真实工作分布
    

---

# 4 评估指标定义

## 4.1 主指标

- `mAP@0.5` 或 `mAP@[0.5:0.95]`
    
- Recall
    
- Precision
    

## 4.2 辅助指标

- 按类别 mAP
    
- 按尺度 mAP（small / medium / large）
    
- PR 曲线与阈值敏感性（score threshold sweep）
    
  

---

# 5 分阶段评估流程

## 5.1 Stage A：PyTorch（基线评估）

**输入**：`.pth`  
**推理精度**：FP32  
**输出**：

- `metrics_pytorch.json`
    
- `detections_pytorch.json`
    
- `config_eval.yaml`（阈值、NMS、输入尺寸、类别映射）
    
- `regression_dump/`（回归集可视化与中间结果可选）
    

**作用**：全链路基线，后续对齐目标

---

## 5.2 Stage B：ONNX（导出一致性评估）

**输入**：`.onnx`  
**推理引擎**：ONNX Runtime  
**输入形式**：RGB（同 Stage A）  
**要求**：

- 同一输入与后处理
    
- mAP 应 **几乎不下降**
    
- Regression Set 的逐图结果应高度一致
    

**重点检查**：

- raw 输出 tensor 数值接近
    
- bbox 数量、位置、score 排序是否一致
    

**若掉点**：  
优先排查：

- 前处理差异
    
- sigmoid / reshape / transpose / opset 行为差异
    
- 输出解码逻辑
    

---

## 5.3 Stage C：BC（编译模型评估）

**输入**：`.bc`  
**输入形式**：YUV/NV12（前处理入图）  
**特征**：可能发生算子替换、融合、精度变化、量化

### 5.3.1 Stage C 的评估拆分（推荐，便于定位）

- **C1：功能与语义对齐验证（Regression Set）**
    
    - 逐图对比：bbox 数量、IoU 匹配、score 分布
        
    - 输出 FN/FP 变化统计
        
    - 目标：证明“输入语义 + 坐标映射 + 后处理”整体正确
        
- **C2：主指标评估（Primary Eval Set）**
    
    - 输出：`metrics_bc.json` + 分类/尺度细分指标
        
    - 目标：量化“编译+融合+量化+前处理入图”的综合影响
        

**评估重点**：

- 是否出现系统性 Recall 下降（量化/阈值/后处理一致性问题）
    
- 是否某一类别或尺度掉点明显（小目标常见）
    

---

## 5.4 Stage D：HBM（最终部署评估）

**输入**：`.hbm`  
**推理方式**：真实 SDK / 真实输入路径（YUV/NV12）  
**这是最终交付精度与上线判定依据**

必须输出：

- `metrics_hbm.json`
    
- `detections_hbm.json`
    
- `latency_profile.json`（端侧时延/吞吐/内存）
    
- `sdk_io_contract.md`（输入格式、stride、量化参数、阈值、版本信息）
    

必须满足：

- 业务定义的 mAP / Recall 门槛
    
- 关键类别不出现不可接受的漏检
    

---

# 6 精度对齐与掉点诊断方法

## 6.1 输出对齐分析（逐图）

对同一张图片：

- 对比 bbox 数量
    
- IoU 匹配后的位置偏移统计
    
- score 分布变化
    
- FP / FN 增减情况
    
- 按尺度（small/medium/large）切分统计
    

> ✅ 当 Stage C/D 输入从 RGB 变成 YUV/NV12 时，逐图对齐是“证明语义一致”的硬证据。

---

## 6.2 常见掉点来源速查表

|现象|常见原因|
|---|---|
|小目标 mAP 掉|padding / stride / 插值差异、letterbox 回映射|
|Recall 明显下降|量化校准不充分、阈值/NMS 实现差异|
|框抖动|浮点精度变化、exp/sigmoid 数值敏感、融合后误差放大|
|类别错乱|输出顺序 / class id 映射错误|
|mAP 波动大|NMS 实现差异、top-k/阈值不一致|

# 7 放行（Gate）机制

## 7.1 阶段放行规则（示例）

- Stage A → Stage B（PyTorch → ONNX）：
    
    - `ΔmAP ≈ 0`（通常要求极小，比如 ≤ 0.1 视任务定）
        
    - Regression Set 逐图一致性通过
        
- Stage B → Stage C（ONNX → BC）：
    
    - `ΔmAP ≤ 阈值`（量化/编译允许小幅下降，但必须可解释）
        
    - 不允许关键类别出现不可接受 Recall 崩塌
        
- Stage C → Stage D（BC → HBM）：
    
    - 部署态指标达标
        
    - 延迟/功耗/内存满足车规/平台约束
        

## 7.2 以 HBM 指标为最终裁决

> 模型选型与上线，只看部署态表现（Stage D）

# 8 评估结果管理与自动化

## 8.1 自动化建议

提供统一 `eval.py` / `eval.sh`：

- 参数指定推理后端（pytorch/onnx/bc/hbm）
    
- 一键跑全链路评估
    
- 自动生成对比表与报告（markdown + json）
    
- Regression Set 失败样本自动导出可视化
    

建议目录规范：

- `outputs/<backend>/<date_tag>/metrics.json`
    
- `outputs/<backend>/<date_tag>/detections.json`
    
- `outputs/<backend>/<date_tag>/regression_viz/`
