# 2 核心对象与职责

## 2.1 Image
**定义**：训练环境的可复用构建产物（只读文件系统层 + 元数据）。  
**存放位置**：镜像仓库 Registry（集中存储） 
**绑定关系**：
- ✅ 绑定：版本（tag/sha）、依赖库、运行入口
- ❌ 不绑定：物理节点、IP、GPU

**关键结论**：换节点只需拉取同一镜像版本（或命中缓存），无需重建。

---

## 2.2 Job / Task（训练作业）
**定义**：对一次训练运行的“声明式描述”：我要什么环境、什么资源、怎么启动、数据在哪里。  
**典型字段**：
- Image: `repo:tag` / `sha256:...`
- Command/Args/Env
- Resources: GPU/CPU/MEM
- Storage: 数据卷引用（PVC / NAS / Turbo）+ 访问权限
- Networking: 端口、服务发现、出口策略
- Identity/Security: ServiceAccount/Token/Secret（平台不同名字不同）

**绑定关系**：
- ✅ 绑定：资源需求与约束、镜像引用、存储引用、权限与密钥
- ❌ 不绑定：具体跑在哪台物理机（由调度器决定）

---

## 2.3 Scheduler（调度/编排层）
**定义**：负责把 Job 放到合适节点上运行，并完成运行前的“装配”。  
**主要工作**：
1. 节点选择：满足 GPU/CPU/MEM、队列、亲和性等约束
2. 资源隔离：cgroup、device plugin、quota（不同平台实现不同）
3. 镜像拉取：从 Registry 拉取镜像层（或命中本地缓存）
4. 存储挂载：把 PVC/NAS/Turbo 挂到容器路径（如 `/data`）
5. 网络配置：创建网络命名空间，分配 IP/网卡，配置路由/NAT/策略
6. 启动容器：把进程启动起来并监控生命周期

**关键结论**：作业“绑定的是声明”，运行位置由调度器动态决定。

---

## 2.4 Node（计算节点：物理机/VM/裸金属）
**定义**：提供实际算力的宿主资源池（GPU/CPU/内存/本地盘/网卡）。  
**绑定关系**：
- ✅ 绑定：硬件资源实际存在于节点
- ❌ 不绑定：镜像、数据、作业定义（都在外部系统/抽象层）

---

## 2.5 Storage（NAS / Turbo / PV/PVC / Object Store）
**定义**：与计算解耦的持久化数据系统。  
**两种常见形态**：
- **文件系统挂载型**：NAS/Turbo/NFS/并行文件系统等，表现为容器内路径（`/data`）
- **对象存储型**：S3/OSS/COS 等，通过 SDK/HTTP 访问（更彻底的解耦）

**绑定关系**：
- ✅ 绑定：数据本体、命名空间（bucket/path/volume id）、访问权限
- ❌ 不绑定：计算节点（任何节点只要网络可达 + 有权限即可访问）

---

## 2.6 Networking（网络与 IP）
**定义**：为容器提供可通信的网络身份与隔离环境。  
**IP 的常见来源**（不同平台不同）：
- Pod/容器 IP：属于虚拟网络命名空间，不是物理机 IP
- Node IP：节点本身的管理/数据网 IP
- NAT 出口 IP：对外访问时看到的公共/出口 IP（可能固定也可能变化）
- Service/VIP：稳定访问入口（服务发现），与具体 Pod IP 解耦

**关键结论**：看到一个 IP 并不意味着绑定某台物理机，它可能只是虚拟网络身份或出口地址。

---

# 3 “解绑”的本质：引用 + 动态装配

## 3.1 计算与环境（镜像）解耦
- 镜像放在 Registry，作业只引用镜像版本
- 节点只是运行时载体：拉镜像 → 起容器 → 跑训练
- 换节点不需要重建镜像，只需要拉取/缓存命中

### 3.1.1 计算与数据解耦
- 数据放在独立存储系统（NAS/Turbo/对象存储）
- 作业通过“卷引用/路径/凭证”声明要用哪些数据
- 调度器在启动时把数据挂载到容器路径或注入访问凭证

### 3.1.2 运行实例可重建，状态要外置
- 容器实例是一次性的运行载体，可能失败、重启、迁移
- 需要持久化的状态（checkpoint、日志、指标）应写入：
  - 持久卷（PVC/NAS/Turbo）
  - 或对象存储（更强解耦）

---

## 3.2 典型训练链路（从提交到运行）
1. 用户提交 Job：镜像引用 + 启动命令 + 资源需求 + 数据引用
2. 调度器挑选节点：满足 GPU/CPU/MEM、队列与约束
3. 节点拉镜像：从 Registry 拉取镜像层（或命中缓存）
4. 挂载数据卷：将 NAS/Turbo/PVC 挂到容器 `/data`
5. 配置网络：分配虚拟网卡与 IP，配置服务发现/出口策略
6. 启动训练进程：容器内执行 entrypoint/command
7. 输出落盘：checkpoint/日志/metrics 写到持久存储或对象存储
8. 作业结束：释放节点资源，持久数据仍保留在外部存储

---

## 3.3 常见误区澄清
- 误区 A：分配了 IP 就是绑定物理机  
  **澄清**：可能是 Pod IP / NAT 出口 IP / Service 地址，网络身份可虚拟化。
- 误区 B：换节点就要重建容器  
  **澄清**：镜像在 Registry，换节点只需 pull（或缓存命中），无需 build。
- 误区 C：容器内写的结果天然会保留  
  **澄清**：写到容器可写层通常不持久；应写到挂载卷或对象存储。

---

## 3.4 实践建议（保证可复现与可迁移）
- 镜像：使用不可变版本（推荐 sha 或严格 tag 管理），并推送到统一 Registry
- 代码与配置：参数化/配置化，避免手动进容器修改
- 数据与产物：数据只读挂载；checkpoint/日志写持久卷或对象存储
- 监控与可观测性：标准化日志路径、指标上报与追踪（run_id/job_id）

---

## 3.5 一句话总结
云端训练的抽象是：**“作业声明 + 动态调度装配 + 外置持久存储”**，从而实现环境、计算与数据的解耦与可迁移。
