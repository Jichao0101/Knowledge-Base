# 1 原始数据集

## 1.1 理论数据量

睁闭眼数据集来自56个场景，每个场景为光线/位置/遮挡的组合，包含
- 常规场景12个，每个场景24个id，每个id执行14个动作，单id抽帧 $14 * 20 = 280$帧
- 复杂场景44个，每个场景2个id，每个id执行14个动作，单id抽帧$14 * 20 = 280$帧
总数据量 $（12 * 24 + 44 * 2）* 280 = 105280$

# 2 构建数据集

## 2.1 构建原则

- 按 **8:1:1** 构建 **训练集：验证集：测试集**
    
- 评估必须可复现：同一版本数据集 + 同一 seed + 同一 split 策略 + 同一输入生成策略（ROI/letterbox） ⇒ 导出结果一致
    
- **样本粒度：单眼**
       
- 数据目录结构不做强假设：以 `annotation/**/BlinkDetection/` 为准进行检索

	- `DMS` 目录名不固定
	    
	- 可能存在多个 BlinkDetection 路径
	    
	- BlinkDetection 下图片与 JSON 可能同级或更深层嵌套
## 2.2 采样方案

### 2.2.1 随机采样（仅用于工程联通验证）

随机切分存在数据泄漏风险（同人/同场景/连续帧），指标偏乐观，仅用于快速验证链路：

- 以 **图片为单位**随机划分 train/val/test（8:1:1）
    
- **同一张图片的左右眼样本必须同 split**（避免“同图泄漏”）
    
### 2.2.2 分组采样

为避免同人/同视频段/同场景泄漏，默认使用 **group split**：

- 从文件名/路径/JSON 中抽取 `person_id`，构成 `group_id`
    
- **按 group_id 分配 split（8:1:1）**，并广播到该 group 下的所有图片与双眼样本
    
- 输出 `split_report.json` / `leakage_audit.txt`：保证 `group_id` 不跨 split（必须为 0）

## 2.3 构建流程

### 2.3.1 Stage A：扫描源目录，生成“可复制清单 + source_manifest”

**目标**：低 I/O 找全 JSON，并行做 JSON 与 bbox 语义校验；同时为后续“按人 split”准备 group_key 统计证据
**输入**：`annotation_root`（只读）  
**扫描规则**：单线程 `os.walk`，只在命中 `BlinkDetection` 子树时收集 `*.json`

**并行检查**（多线程，不读图片）：

- JSON 能否解析
    
- 是否有 `filePath`（字符串）
    
- 是否有 `info.width/height`（若无则 oob=unknown，但不拒绝）
    
- `dataList` 是否为 list 且可展开
    
- 对每个 `dataList[i]`：
    
    - `properties.eye_status` ∈ 合法集合
        
    - `coordinates` 形状为 `[[x1,y1],[x2,y2]]`
        
    - x1,y1,x2,y2 可转 float，且非 NaN/Inf
        
    - x2>x1, y2>y1
        
    - (x2-x1),(y2-y1) ≥ min_size
        
    - **越界检查**：如果 json 里有 width/height，则检查 0<=x1<x2<=W, 0<=y1<y2<=H；如果没尺寸信息，越界检查记为“unknown”但不拒绝

- `group_key_guess`：从 json_rel / filename tokens 中提取候选 person（不做强假设，只输出候选与命中规则版本）

**产出**：

1. `blink_roots.txt`（命中的 BlinkDetection 目录绝对路径）
    
2. `source_manifest.json`（json_rel/img_rel_guess/filePath/samples/校验统计 + group_key_guess）
    
    - `img_rel_guess：`json_rel_parent + "/" + filePath`（相对 annotation root）
        
3. `data_checklist.txt`（缺图率在 A 阶段无法算“真实缺图率”，只能算“filePath 字段缺失率/无法推导率”）
4. `group_key_stats.json`：缺失率、唯一值数量、topK 频次、长尾分布（用于判断这个 key 是否真的是 person_id）
    
> 关键点：Stage A 不碰图片文件，所以不会受到权限/依赖限制；但“missing image”只能在后续阶段确认。
> 注意：不假设存在 `DMS/`，也不要求 `BlinkDetection` 必须位于固定层级  
> 只要它在 `annotation/**/BlinkDetection/**` 命中，就保留原始相对结构

### 2.3.2 ### Stage B 云桌面复制数据到私人目录

**输入**：`source_manifest.json` 
**动作**：

- 对每条：
    
    - 复制 json 到 `DEST/<json_rel>`
        
    - 推导图片路径：`img_abs = json_abs.parent / filePath`
        
        - 如果存在就复制到 `DEST/<img_rel_guess>`（即同相对路径）
            
        - 如果不存在：记录到 `missing_images.txt`
            
**产出**：

- `missing_images.txt`
    
- `copy_report.json`（拷贝统计、失败样例）
    
- `copied_jsons.txt` / `copied_images.txt`（便于抽检）

**使用云桌面而不是容器**：

- 云桌面更适合做大规模 I/O。
    
- 保持相对路径不变，容器只需要整体拷贝 `DEST` 目录即可。

### 2.3.3 Stage C：容器内升级 final_manifest（读图校验 + split + build）

#### 2.3.3.1 样本粒度与 split 粒度

- 样本粒度：单眼（每条 json 的 dataList 展开）
- split 粒度：**group 级**（person_id）
- split 比例：8:1:1
- seed 固定
#### 2.3.3.2 数据检查

- 再次解析 json（防复制过程损坏）
    
- **读图**拿真实 W/H
    
- bbox 越界按真实 W/H 判定
    
- `img_path/json_path` 写为 **dest 相对路径**
    
- `image_id = sha1(img_rel)` 绑定同图双眼

- `group_id = f(person_id)`
    
- 基于 group_id 生成 split，并广播到 group 内所有样本
    

输出：
    
- `final_checklist.txt`：真实缺图率、读图失败率、bbox 越界率、label 分布、split 分布
    
- `split_report.json`：按 group_id 统计各 split 的 group 数/样本数
    
- `leakage_audit.txt`：检查同一 group_id 是否跨 split（必须为 0）
    
#### 2.3.3.3 manifest 升级字段

`build`总字段：
	`manifest_version`
	`seed`
	`split_ratio`
	`split_policy`
	`filter_rule`
	`timestamp`
	`label_dist_per_split = {split: Counter(label)}`
    `group_dist_per_split = {split: n_groups}`
    `sample_dist_per_split = {split: n_samples}`

每条样本包含：

- `img_path`：dest 相对路径
    
- `json_path`：dest 相对路径
    
- `image_id`：sha1(img_rel)
    
- `group_id`：person id
    
- `eye_id`：来自 json item.id
    
- `bbox_gt`：`[x1,y1,x2,y2]`（float）
    
- `label`：eye_open/eye_closed/eye_occluded/eye_narrow
    
- `split`：train/val/test
    
- `meta`：结构化信息（scene/camera/pose/illumination）

### 2.3.4 Stage D：离线导出 ROI/Crop ImageFolder

#### 2.3.4.1 目标与边界

该阶段用于固化训练/评测输入，支持后续输入方案对照实验。  
输出不仅包含传统 **ROI + padding** 方案，也支持 **中心扩展 crop** 方案。

目标：

- 快速训练 / 验证
    
- 固定 bbox 扰动分布，保证可复现
    
- 将“输入生成策略”与“训练增强策略”解耦
    
- 支持多种输入构造方式对比
    

不在 Stage D 内执行：

- resize 后颜色增强
    
- 随机擦除
    
- 最终 normalize
    
- 模型相关增广
    

这些统一在训练代码内完成。

## 2.5 输入生成策略

### 2.5.1 ROI + Padding 方案

基于 `bbox_gt` 裁剪眼部 ROI，再进行 keep-ratio resize 与 padding 到方形。  
适合作为传统基线方案。

候选 padding 方式：

1. `reflect / replicate`
    
2. `constant 0`
    
3. `constant gray`
    

### 2.5.2 Eye-Center Expand Crop 方案

基于标注框计算眼睛中心 `(cx, cy)`，再从原图直接裁取正方形上下文区域：

crop_size = k * max(w_bbox, h_bbox)

其中：

- `k` 为上下文扩展倍率
    
- 推荐候选：`1.4 / 1.8 / 2.2`
    

该方案特点：

- 不再向图像中引入人工 padding 纹理
    
- 保留更多真实眼周上下文
    
- 更适合 `eye_occluded` 等依赖环境与遮挡上下文的类别

#### 2.5.2.1 Bounding Box 离线增强策略

##### 2.5.2.1.1 核心原则

- bbox 增强发生在裁剪 ROI 之前，用于模拟推理时“关键点/检测框 → 外接框”的偏差
- 训练集随机扰动；验证/测试集固定规则（可复现）

##### 2.5.2.1.2 训练集jitter（随机）

对 `B_gt` 随机扰动得到 `B_aug`，再裁剪：

- 对称外扩/内缩：`pad_ratio ∈ [-0.10, 0.15]`
- 平移：`dx, dy ∈ [-0.03, 0.03] × bbox_size`
- 约束：最小宽高 `min_size` + clamp 到图像边界
	- **不重采样**：失败则丢弃该 aug，并在统计里记录 `n_aug_dropped_small/n_aug_dropped_oob`

命名包含 `aug_id`：

- `{image_id}__{eye_id}__aug{0..K-1}__{label}.jpg`

随机数不允许用“全局 random.seed + 多线程”的隐式顺序，因为多线程会导致随机序列消费顺序变化，使用 **样本级确定性随机**：

- `rng_seed = sha1(seed, image_id, eye_id, aug_id)` → 生成 pad/dx/dy

##### 2.5.2.1.3 验证/测试 jitter（固定，两套可选）

验证/测试必须稳定可复现，可输出两套评估口径，目前使用PAD

1) **部署偏差评估（PAD）**  
- 固定外扩：`pad_ratio = 0.15` 得到 `B_eval`

2) **理想上限评估（GT）**  
- 不外扩：`pad_ratio = 0.0`（或直接 bbox_gt）

#### 2.5.2.2 导出索引（roi_manifest.json）

生成 `roi_manifest.json`，支持从 ROI 回查原图：

`build`字段：`manifest_version`, `seed`, `split_ratio`, `filter_rule`,`bbox_aug_cfg`

对于每个roi：
- `roi_path`
- `img_path`
- `json_path`
- `image id`
- `eye_id`
- `pad_ratio, dx_ratio, dy_ratio`
- `rng_key = f"{seed}::{image_id}::{eye_id}::{aug_id}"`
- `bbox_used`（实际裁剪框）
- `bbox_gt`
- `split`
- `label`
- `meta`

## 2.6 数据追溯与误分类分析机制

### 2.6.1 设计目标

为保证模型训练与评估结果具备可解释性与可审计性，系统需支持从任意一个 ROI 样本或误分类结果，逐级回溯至其对应的：

- ROI 文件
    
- ROI 导出配置
    
- 原始图片
    
- 原始 JSON 标注
    
- 原始标注框
    
- 实际裁剪框
    
- 数据场景元信息
    

该机制用于支持以下分析任务：

- 检查误分类样本是否存在标签错误
    
- 检查某类误分类是否集中在特定场景、相机位、光照或配饰条件下
    
- 检查 ROI 导出策略（padding / center crop / jitter）是否引入异常分布
    
- 检查数据分布是否存在长尾或结构性偏差
    

---

### 2.6.2 追溯原则

#### 2.6.2.1 单样本可追溯

对于任意一个训练、验证或测试样本，必须能够唯一定位其来源。  
即：

ROI 样本 → roi_manifest 条目 → 原始图片 / 原始 JSON / bbox_gt / bbox_used / meta

#### 2.6.2.2 误分类可追溯

对于测试集中每一个误分类样本，必须能够导出其完整追溯信息，以支持后续人工抽检和脚本统计分析。

#### 2.6.2.3 结果与数据版本绑定

避免不同版本之间出现样本路径相同但语义不一致的情况。

---

### 2.6.3 ROI 索引机制

#### 2.6.3.1 索引来源

Stage D 导出 `roi_manifest.json`，作为 ROI 样本与原始样本之间的主索引文件。

#### 2.6.3.2 索引键

每个 ROI 样本以 `roi_path` 作为唯一索引键

---

### 2.6.4 测试阶段追溯输出

#### 2.6.4.1 输出文件

测试阶段统一输出以下文件：

test_trace_all.jsonl  
test_trace_misclassified.jsonl  
test_trace_misclassified.csv  
test_trace_summary.json

### 2.6.5 误分类聚合分析

#### 2.6.5.1 混淆对统计

对所有误分类样本，按 `(gt_label, pred_label)` 聚合统计，得到主要混淆对，例如：

- `eye_open -> eye_narrow`
    
- `eye_closed -> eye_open`
    
- `eye_occluded -> eye_open`
    

该统计用于识别主导错误模式，而非仅依赖整体 confusion matrix。

#### 2.6.5.2 场景分布分析

基于 `meta.tokens` 或结构化场景字段，对误分类样本进一步按以下条件聚合：

- camera
    
- scene
    
- illumination
    
- weather
    
- accessory
    
- pose
    
- seat / person_id
    

用于分析误分类是否集中在某一类真实场景条件下。

#### 2.6.5.3 导出策略分析

基于以下字段分析 ROI 导出策略是否影响识别结果：

- `pad_ratio`
    
- `dx_ratio`
    
- `dy_ratio`
    
- `bbox_used / bbox_gt` 偏差
    

若误分类集中于大偏移、大外扩或特定 crop 模式，则说明问题可能来自输入构造，而非模型主干能力不足。

---

### 2.6.6 标注质量排查规则

追溯机制需要支持对疑似脏标签样本进行筛选。  
建议优先检查以下样本：

#### 2.6.6.1 高置信误分类样本

满足：

- `correct = false`
    
- `confidence >= 0.9`
    

该类样本可能存在：

- 标签错误
    
- ROI 导出错误
    
- 原始框错位
    

#### 2.6.6.2 边界类别样本

满足以下混淆对的样本优先抽检：

- `eye_open -> eye_narrow`
    
- `eye_narrow -> eye_closed`
    
- `eye_closed -> eye_narrow`
    

该类样本通常反映：

- 类别边界定义模糊
    
- 标注标准不一致
    
- 眼部状态过渡帧定义不清
    

#### 2.6.6.3 遮挡类异常样本

对于 `eye_occluded` 的误分类样本，应重点检查：

- ROI 是否裁到真实遮挡区域
    
- 是否需要更大上下文
    
- 标签是否把“眯眼”与“遮挡”混用

# 3 模型结构

- 采用 **MobileNetV3-Small** 作为基础网络
    
- 使用 ImageNet 预训练权重进行初始化
    
- 修改最后分类层为 **4 分类**：`eye_open=0, eye_closed=1, eye_occluded=2, eye_narrow=3`

# 4 模型输入与数据增强

## 4.1 训练

训练侧建议保持“输入生成策略”和“在线增强策略”分层：

1. Stage D 生成基础输入（ROI 或 CenterCrop）
    
2. 训练时执行轻量在线增强
    

在线增强建议：

1. `resize_keep_ratio(long=128)` 或 `resize(128,128)`
    
    - 取决于输入方案
        
2. `pad_to_square(128)`（仅 ROI 方案需要）
    
3. `RandomHorizontalFlip`
    
4. `ColorJitter`
    
5. `RandomErasing`
    
6. `ToTensor`
    
7. `Normalize`
    

## 4.2 验证 / 测试

1. 与训练保持相同输入构造逻辑
    
2. 不使用随机增强
    
3. 保证同版本、同 seed、同配置下结果可复现

# 5 训练配置（Baseline）

- 输入尺寸：128 × 128
    
- Batch size：64
    
- Optimizer：AdamW
    
    - learning rate = 1e-3
        
    - weight decay = 1e-4
        
- Loss：CrossEntropyLoss
    
    - 可选 label smoothing = 0.0
        
- Scheduler：CosineAnnealingLR
    
- Epoch：100
    
- AMP：开启
	 
- EMA：开启

- DDP 关闭
	
- 指标：
    - Macro-F1
    - Accuracy@1
    - Precision/Recall


# 6 实验对照设计

## 6.1 设计目标

针对当前模型在 `eye_narrow` 与 `eye_occluded` 类别上的识别不足，设计可复现的对照实验，从以下两个维度定位主因并评估优化收益：

- **Loss 设计**：评估类别不平衡与难样本学习问题对模型性能的影响
    
- **输入构造方式**：评估 padding 伪特征与真实上下文保留方式对模型性能的影响+
- 
    

实验设计遵循单变量控制原则，避免多个因素同时变化导致结论不可归因。

## 6.2 实验原则

所有对照实验需满足以下一致性要求：

- 使用同一版本数据集
    
- 使用同一 `split` 文件与 `seed`
    
- 使用同一 backbone、优化器、学习率策略与训练轮数
    
- 使用同一评估代码与指标统计方式
    
- 除被比较变量外，其余训练与评估配置保持一致
    

所有实验统一输出以下结果：

- `Macro-F1`
    
- `Accuracy@1`
    
- `Macro-Precision`
    
- `Macro-Recall`
    
- 各类别 `Precision / Recall / F1`
    
- confusion matrix
    
- 训练日志与最终 checkpoint
    

其中，**Macro-F1 为主指标**，并重点关注：

- `eye_narrow` 的 Recall / F1
    
- `eye_occluded` 的 Recall / F1
    
- `eye_open` 的 Precision 是否显著下降
    
- confusion matrix 中关键误分类迁移方向是否改善

## 6.3 Loss 对照实验

### 6.3.1 实验目标

验证当前性能瓶颈是否主要来自类别分布不均衡，并评估不同 loss 设计对小类识别能力的改善效果。

### 6.3.2 控制变量

固定以下配置不变：

- 输入方案：`ROI + reflect/replicate padding`
    
- backbone：`MobileNetV3-Small`
    
- 数据增强、优化器、学习率策略、训练轮数与 batch size
    

仅调整 loss 类型。

### 6.3.3 实验分组

#### 6.3.3.1 A1 基线方案：CrossEntropyLoss

作为默认训练基线，用于衡量后续改动收益。

#### 6.3.3.2 A2 类别加权交叉熵：Weighted CrossEntropyLoss

根据训练集类别频次设置权重，采用以下形式：

weight_c ∝ 1 / sqrt(freq_c)

该方案用于缓解高频类别对整体梯度的主导作用。

#### 6.3.3.3 A3 Focal Loss

采用 focal loss 抑制易分类样本对总损失的贡献，增强对难样本的学习能力。初始配置为：

gamma = 2.0

### 6.3.4 结果判定原则

- 若 `Weighted CE` 已显著提升 `eye_narrow` 与 `eye_occluded`，则说明当前主因主要为类别不平衡
    
- 若 `Weighted CE` 效果有限，而 `Focal Loss` 或 `Weighted Focal Loss` 继续带来提升，则说明难样本学习不足也是主要因素
    
- 若 `Focal Loss` 提升小类 Recall，但显著降低大类 Precision，则说明其对边界模糊样本或噪声样本较为敏感
    

### 6.3.5 实验顺序

按以下顺序推进：

CE -> Weighted CE -> Focal Loss

## 6.4 构造对照实验

### 6.4.1 实验目标

评估不同输入构造方式对分类性能的影响，重点分析：

- padding 是否引入伪特征
    
- 真实上下文是否有助于 `eye_occluded` 识别
    
- 输入构造方式是否影响 `eye_narrow` 与 `eye_open / eye_closed` 的边界区分
    

### 6.4.2 控制变量

固定以下配置不变：

- loss：使用 loss 对照实验中的最优方案；若尚未完成，默认采用 `Weighted CE`
    
- backbone、优化器、学习率策略、训练轮数与评估方式保持一致
    

仅调整输入生成策略。

### 6.4.3 实验分组

#### 6.4.3.1 B1 基线方案：ROI + reflect / replicate padding

保持当前输入构造流程，作为输入侧对照基线。

 优点

- 保持眼睛几何比例
    
- 不会像直接 resize 到 128×128 那样把眼睛横向/纵向拉变形
    
- 对分类任务一般比强行拉伸稳
    

 缺点

- padding 区域不是自然图像内容
    
- reflect 可能制造“假睫毛/假眼角/假高光”式镜像纹理
    
- replicate 会制造整片平直边界，卷积很容易感知出来
    
- 当 ROI 很扁时，模型可能学到“有效区域占比”而非眼睛状态

#### 6.4.3.2 B2 对照方案：ROI + constant 0 padding

使用常数 0 进行 padding，用于评估 reflect / replicate 是否确有收益，并验证显式黑边是否形成 shortcut。

优点

- 实现最简单
    
- 不引入镜像伪纹理
    
- 量化部署最友好，NPU/ISP 也最容易对齐
    

缺点

- 黑边是非常强的人工模式
    
- 模型极容易把边框面积/位置当 shortcut
    
- 如果数据里 bbox aspect ratio 与类别有相关性，会形成伪相关

#### 6.4.3.3 B3 对照方案：ROI + constant gray padding

使用固定灰度值进行 padding，采用：`114`

该方案用于降低镜像伪纹理，同时评估常数 padding 对模型的影响。

优点

- 比纯黑边柔和一点
    
- 比 reflect 少了假纹理
    
- 与很多检测/分类部署链路兼容性较好
    

 缺点

- 依然是人工常数区域
    
- 仍然可能泄露有效区域占比
    
- 灰值选取不当时，会形成非常稳定的背景模式
    

#### 6.4.3.4 B4 候选优化方案：Eye-Center Expand Crop

基于眼框中心直接在原图裁取正方形上下文区域，避免人工 padding。  
裁剪尺度定义为：

crop_size = k * max(w_bbox, h_bbox)

候选倍率：

- `k = 1.4`
    
- `k = 1.8`
    
- `k = 2.2`

 优点

1. **消除 padding artifact**
    
    - 不再往图里硬塞假背景
        
2. **保留真实上下文**
    
    - 对 `occluded` 很关键，遮挡往往不是只发生在眼裂内部，而是在眼周
        
3. **输入分布更接近部署**
    
    - 如果部署也是关键点/框中心 + 固定窗口，这个方案更一致
        
4. **减少长宽比捷径**
    
    - 不再让“有效区域占比”成为强线索
        

缺点

1. **引入更多无关背景**
    
    - 包括眉毛、鼻梁、皮肤、镜框、发丝、环境光
        
    - 如果 crop 太大，模型会学偏
        
2. **依赖中心估计稳定**
    
    - 中心偏了，眼睛可能不在合适位置
        
    - 对大 yaw、大遮挡更敏感
        
3. **尺度选择变成新超参**
    
    - crop 太小：上下文不足
        
    - crop 太大：背景污染
        
4. **可能降低 open/closed 的纯局部判别效率**
    
    - 因为输入里掺了更多杂质

### 6.4.4 结果判定原则

- 若 `ROI + constant 0/gray padding` 明显劣于 `ROI + reflect/replicate`，说明保持几何比例与边缘连续性更重要
    
- 若 `CenterCrop` 显著提升 `eye_occluded` 的 Recall / F1，说明真实上下文为关键判别信息
    
- 若 `CenterCrop` 使 `eye_open / eye_closed` 性能显著下降，则说明 crop 尺度过大，引入了过多背景噪声，需要收缩 `k`
    

### 6.4.5 推荐实验顺序

按以下顺序推进：

ROI_Reflect -> ROI_Const0 -> ROI_ConstGray -> CenterCrop(k sweep)

其中 `CenterCrop` 为重点优化方向，评估 2~3 个尺度倍率。

## 6.5 联合实验设计

在 Loss 对照实验与输入构造对照实验分别完成后，再进行联合优化验证。

评估以下组合：

- `CE + ROI_Reflect`：当前基线
    
- `Weighted CE + ROI_Reflect`：类别平衡基线
    
- `Weighted CE + CenterCrop(k)`：输入优化方案
