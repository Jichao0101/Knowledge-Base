# 1 查找调试器

1. 查找gdb
```
find / -type f -name 'gdb' 2>/dev/null
```

- `find /`：从根目录 `/` 开始递归扫描（全盘搜）。
    
- `-type f`：只要“普通文件”（避免把目录/设备节点也列出来）。
    
- `-name 'gdb'`：文件名**完全等于** `gdb`（不是包含 gdb 的意思）。
    
- `2>/dev/null`：把错误输出丢掉。  全盘搜会遇到很多“Permission denied”，从 stderr（fd=2）重定向到黑洞设备

2. 查找交叉gdb
```
find / -type f -name 'aarch64*-gdb' 2>/dev/null
```

# 2 Coredump调试

1. 目标机（J6M）崩溃时生成 core
    
2. 把 core 文件拷回 PC
    
3.  PC 上用 gdb 打开： 
```
aarch64-none-linux-gnu-gdb /path/to/sdk /path/to/core 
(gdb) bt 
(gdb) bt full
```
`sdk`：**本地的可执行文件**（必须和板端同一版本/同一构建产物），用于读取符号和源码映射

> [!NOTE] 
> export LD_LIBRARY_PATH 是在 **程序启动时** 用来找 so 的，此时程序已经死亡，对 core 调试是无效的，需要设置sysroot 或solib-search-path

  
# 3 远程在线调试

[[远程调试]]
## 3.1 PC 交叉 gdb + 板端 gdbserver

- **gdb（PC/容器上）**：负责解析符号、显示源码、下断点、看变量、执行调试命令。
    
- **gdbserver（J6M 上）**：负责把正在运行/即将运行的进程挂到调试协议上

## 3.2 板端启动gdbserver
### 3.2.1 拷贝gdbserver到板端

 ```
 scp <src> root@<J6M_IP>:<dst>
 chmod +x gdbserver
 ```

### 3.2.2 gdbserver启动并托管程序

```
./gdbserver :2345 ./sdk
```

## 3.3 本地容器启动gdb

### 3.3.1 启动gdb

```
/bin/aarch64-none-linux-gnu-gdb ./sdk
```
- 这个 gdb 是跑在 x86_64 的，但它“懂 aarch64 目标”，能解析 aarch64 寄存器、调用约定等。
    
- `./sdk`：**本地的可执行文件**（必须和板端同一版本/同一构建产物），用于读取符号和源码映射

### 3.3.2 连接gdbserver

```
(gdb) target remote <J6M_IP>:2345
```

- target remote：目标不是本地进程，而是远程 stub
    
- <J6M_IP>:2345：连接到板端 gdbserver 的 TCP 端口

### 3.3.3 设置根目录

把板端 rootfs（或 SDK/yocto 生成的 sysroot）放在本地某个目录/path/to/j6m_rootfs

```
(gdb) set sysroot /path/to/j6m_rootfs
```

### 3.3.4 设置动态库查找地址

```
(gdb) set solib-search-path /path/to/j6m_rootfs/lib:/path/to/j6m_rootfs/usr/lib
```

### 3.3.5 调试

```
(gdb) info threads
(gdb) bt
```
