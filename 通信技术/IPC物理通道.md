
# 1 TCP（IP + 端口）

**定位**：跨机器 / 跨进程的**可靠字节流**传输协议，基于 IP 层之上。

**特性：**

- 面向连接（connection-oriented）：
  - 需要三次握手建立连接，断开时四次挥手。
- 可靠性（reliability）：
  - 包丢了会重传，乱序会重排，保证**对上层来说是一个有序、不丢、不重复的字节流**。
- 字节流（stream）：
  - 不关心“消息”的边界，上层需要自己定义消息格式（长度 + 内容等）。
- 支持跨主机通信：
  - 通过 IP + 端口进行寻址：`192.168.1.10:5000`。

**优点：**

- 通用、标准、跨平台、跨机器。
- 有内核级拥塞控制和可靠传输，编程模型简单（socket）。

**缺点：**

- 相对本机 IPC（如 shm）开销更大（协议栈、拷贝更多）。
- 无“消息边界”，要自己做 framing。
- 一般需要明确对端 IP/端口，拓扑变化时配置复杂。

**典型场景：**

- 分布式服务之间的 RPC（自研、gRPC 等）。
- 消息中间件（Kafka、ZeroMQ、CyberRT）底层的传输载体。
- 任意跨主机通信。

# 2 管道（pipe）

**定位**：同一机器上，进程间的**单向字节流** IPC。

常见有两种：

1. 匿名管道（anonymous pipe）：
   - 通过 `pipe()` 系统调用创建，通常用于**父子进程**通信。
   - 一端写、另一端读，通常是单向。
2. 命名管道（named pipe / FIFO）：
   - 有一个路径名（如 `/tmp/my_fifo`），不要求必须是父子进程。
   - 任意进程可以通过这个路径打开读/写。

**特性：**

- 本质也是一种**字节流**，类似本地化的 TCP，但只在同一台机器上。
- 由内核维护缓冲区（ring buffer），读写都经过内核。
- 通常是**半双工 / 单向**（除非你建两条 pipe 分别做两个方向）。

**优点：**

- 简单，系统调用少，适合**父子进程**快速搭一条通信通道。
- 不需要 IP/端口，只要在同一台机器上。

**缺点：**

- 仍然是字节流，没有消息边界。
- 不适合复杂拓扑、多消费者/多生产者场景（需要自己管理）。
- 只能本机，跨机扩展困难。

**典型场景：**

- Shell 管道：`ls | grep xxx` 背后就是匿名 pipe。
- 简单的 “守护进程+worker 子进程” 通信。
- 程序之间的简单命令/控制消息传递。

# 3 共享内存（shm）

**定位**：速度最快的本机 IPC，通过把同一块物理内存映射到多个进程的地址空间中实现**“零拷贝”数据共享**。

**特性：**

- 多个进程共享同一块物理内存区域：
  - System V 共享内存（`shmget/shmat`）
  - POSIX 共享内存（`shm_open/mmap`）
- 内核只负责建立映射，不负责同步：
  - 没有读写队列、没有“谁先谁后”的保障；
  - 需要自己使用信号量/互斥锁/自旋锁等同步原语。

**优点：**

- **速度非常快**：
  - 大块数据传输可以做到几乎零拷贝。
  - 非常适合传输大消息（图像、点云、深度图等）。
- 只要映射关系建立，读写成本几乎等同于普通内存访问。

**缺点：**

- 代码复杂度较高：
  - 需要自己设计 ring buffer / 队列结构；
  - 手动处理同步（锁、原子变量、Barrier 等）。
- 生命周期与清理复杂：
  - 进程异常退出可能留下“孤儿 shm segment”。
- 只能用于本机 IPC，不直接跨机器。

**典型场景：**

- 高性能场景：视频帧、点云数据在不同模块之间传输。
- RT 系统或 HMI 中，多个进程共享大块只读/少写数据。
- 实时数据共享（如共享状态表、缓存）。

# 4 消息队列（Message Queue, MQ）

这里说的是 **OS 层面的消息队列**（如 System V MQ / POSIX MQ），不是 Kafka 这种分布式 MQ。

**定位**：由内核维护的**带消息边界的队列**，提供“消息为单位”的 IPC。

**特性：**

- 按消息收发，不是纯字节流：
  - 一次 `msgsnd()` 发一条消息，一次 `msgrcv()` 收一条。
- 通常支持消息大小上限、优先级等：
  - 可以按类型、优先级获取消息。
- 内核负责排队、缓存、唤醒等待进程。

**优点：**

- 有**消息边界**，不用自己切包、粘包。
- 内核负责队列管理，代码相对简单。
- 一对多/多对多比单纯 pipe 好管理一些。

**缺点：**

- 消息仍需在用户态与内核之间拷贝。
- 队列容量有限，满了需要考虑阻塞/丢弃策略。
- 一般只能本机；跨机时需要配合 TCP/UDP 等。

**典型场景：**

- 传统 UNIX/C 系统中的业务模块 IPC。
- 简单的控制指令、状态通知队列。
- 可靠性要求高但数据量不大的场景。

# 5 UDP（IP + 端口，无连接）

**定位**：跨机器 / 跨进程的**无连接数据报**传输协议，位于 IP 层之上，强调**低延迟、低开销、可观测性**

**特性：**

- 无连接（connectionless）：
    
    - 发送端无需建立连接，直接按 `IP + 端口` 投递数据报。
        
    - 接收端只需监听端口即可接收。
        
- 不可靠传输（best-effort）：
    
    - 不保证送达、不保证顺序、不保证不重复。
        
    - 丢包、乱序由上层自行处理或容忍。
        
- **数据报语义（message-oriented）**：
    
    - 每次 `sendto()` 对应一个完整消息；
        
    - 接收端 `recvfrom()` 以消息为单位接收，**天然有消息边界**。
        
- 支持多种寻址模式：
    
    - **单播（unicast）**：发给一个确定 IP。
        
    - **广播（broadcast）**：发给子网内所有节点（如 `192.168.55.255`）。
        
    - **组播（multicast）**：发给加入同一组播组的节点。
        

**优点：**

- 协议栈简单、延迟低、抖动小。
    
- 无连接、无重传，**行为可预测**，适合实时系统。
    
- 数据报可被 **Wireshark / tcpdump 直接捕获与解析**，调试友好。
    
- 天然支持一对多（广播 / 组播）发布语义。
    

**缺点：**

- 不可靠，需要上层自行容忍或补偿丢包。
    
- 不适合大数据量（易分片、丢包概率高）。
    
- 广播/组播依赖网络配置，跨网段受限。
    

**典型场景：**

- 实时状态、心跳、姿态、状态机数据发布。
    
- 车载 / 工业控制中的 RT 通信。
    
- 调试态的多节点监听与抓包验证。

# 6 SSH 对通信协议的影响说明

## 6.1 现象描述

- 使用 **ZMQ/TCP** 进行 A 核 → PC 通信时：
    
    - 多台 PC 即使监听同一端口，也都能收到数据；
        
    - 抓包时只能看到 **SSH 加密流量**，无法解析应用层消息。


## 6.2 原因解析

**关键点：并不是 TCP 或 ZMQ 具备“广播能力”，而是 SSH 在中间扮演了“隐形转发器”的角色。**

### 6.2.1 TCP + SSH 的真实拓扑

每一台建立 SSH 连接的机器，实际上都拥有一条**独立的 TCP 隧道**：

```
PC_A: localhost:5555 ──SSH──▶ A核:5555 
PC_B: localhost:5555 ──SSH──▶ A核:5555 
PC_C: localhost:5555 ──SSH──▶ A核:5555
```

- ZMQ PUB 端看到的是 **多条 TCP 连接**
    
- 每条连接都会各自收到一份数据
    
- 表现出来就像“广播”
    

但实际上：

> **这是 TCP 的多连接扇出（fan-out），不是网络广播**

### 6.2.2 抓包为何不可见

- TCP 流量被封装在 SSH tunnel 中；
    
- 网络层只能看到：
    
    `TCP → SSH → 加密 payload`
    
- 应用层消息已不可直接解析。
    

---

## 6.3 UDP 与 SSH 的根本区别

UDP 通信为：

`A核 ──UDP──▶ IP:PORT`

- 不依赖连接
    
- 不经过 SSH tunnel
    
- 不会被加密
    
- 抓包工具可直接看到真实 payload
    

因此：

> **UDP 才是真正“裸露在网络层”的通信方式**
