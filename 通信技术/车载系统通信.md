# 1 总体图景：各通信技术的层级关系

## 1.1 逻辑分层图

```
应用层：  智驾/座舱程序、算法模块
         ↑
通信框架：ZMQ / ROS / Cyber RT / 自研中间件
         ↑
数据编码：Protobuf / JSON / 自定义二进制协议
         ↑
传输层：  TCP（面向连接，可靠流） / UDP（无连接，报文）
         ↑
网络层：  IP（寻址、路由）
         ↑
链路层：  ETH（以太网） / CAN / LIN / FlexRay
         ↑
物理层：  双绞线、光纤、电平

```
应用层代码在表示层相关框架的支持下，对消息进行编码和加密，例如使用 Protobuf 进行序列化；随后基于会话层提供的套接字接口，选择传输层的 UDP 或 TCP 协议，将数据通过数据链路层（如以太网）以及物理层（如车载总线）发送给接收端

ZMQ、CyberRT 这类通信框架，本质上是在 socket 之上提供了一层“**以 topic/channel 为中心**”的抽象：开发者只需要关心消息的类型和主题，不需要直接操作 IP 地址、端口号和连接状态。  
如果不使用这类框架，则可以将 Protobuf 等方式编码后的消息，直接通过 UDP/TCP 套接字发送到指定 IP:Port，此时连接管理、错误处理、负载均衡等都需要由业务或自研框架自己承担

# 2 socket：应用程序的“通信插座”
## 2.1 基本概念

- socket 是 OS 内核里的一种对象，对应一个 **文件描述符 fd**。
    
- 应用程序通过 fd 调用 `send/recv` 来收发数据，本质上和读写文件类似。
    

常见创建方式（IPv4）：

```
// TCP 
int sock = socket(AF_INET, SOCK_STREAM, 0);  
// UDP 
int sock = socket(AF_INET, SOCK_DGRAM, 0);  
// CAN（Linux） 
int sock = socket(PF_CAN, SOCK_RAW, CAN_RAW);
```

## 2.2 套接字生命周期（典型 TCP 服务端）

```
socket()  ->  bind()  ->  listen()  ->  accept()  ->  recv()/send()  ->  close()    创建fd       绑定本地    进入监听       接受连接         数据收发         关闭
```

典型 TCP 客户端：

`socket()  ->  connect()  ->  recv()/send()  ->  close()`

UDP 更简单：

`socket()  ->  sendto()/recvfrom()  ->  close()`

# 3 传输通道

## 3.1 TCP：可靠的“字节流”

特点：

- 有连接（三次握手、四次挥手）。
    
- 可靠、有序、不丢不重（除非异常断链）。
    
- 流式：没有消息边界概念（只是一串字节）。
    

适合场景：

- 地图更新、日志上传。
    
- 车端与云端/调度系统的通信。
    
- 有状态控制指令（需要确认、重试）。
    

需要注意的坑：

- **没有消息边界**：必须自己设计协议，解决“粘包/拆包”问题。
    
    - 典型做法：`[固定包头 + 长度字段 + payload]`
        
- `send()` 返回值不一定等于你想发的长度，需循环发送。
    
- 连接断开检测要精心设计（心跳、超时等）。
    

## 3.2 UDP：简单粗暴的“报文”

特点：

- 无连接，发送不确认是否送达。
    
- 按报文（datagram）收发，天然有消息边界。
    
- 速度快、开销小，但丢包、乱序都可能发生。
    

适合场景：

- 激光雷达点云、雷达目标、视觉检测框等高频数据广播。
    
- （公司内部）模块间、ECU 间的实时数据通知。
    
- 不要求每一帧都必须到达，但强调低延迟。
    

典型要点：

- 通过 `sendto/recvfrom` 发送/接收：
    
    - 一次 `recvfrom` 就得到一个完整报文（在 MTU 限制内）。
        
- 丢包容忍度由上层决定：例如追求最新一帧，而不是每帧都到。

## 3.4 智驾场景中的典型选择建议

- **定位数据、感知目标广播**：UDP（追实时性，丢几帧可以接受）。
    
- **OTA 升级、地图下发、配置同步**：TCP（数据必须完整无误）。
    
- **调试工具与车端交互**：一般 TCP（调试命令不适合丢失）。
    
- **大数据块本机传输（图像/点云）**：共享内存 + 轻量通知（如 pipe/MQ/socket）。
    
- **控制类小消息的本机 IPC**：消息队列或本地 TCP/Unix Domain Socket

# 4 通信框架

## 4.1 ZMQ

- 一个高层消息库，封装了底层 TCP / IPC / inproc 等。
    
- 提供常见的通信模式：
    
    - PUB / SUB（发布订阅）
        
    - REQ / REP（请求应答）
        
    - PUSH / PULL（任务分发）
        

用途：

- 简化复杂拓扑：多个模块之间的消息流拓扑用模型表达，而不用自己管所有连接。
    
- 提供自动重连、缓冲队列、队列水位控制等机制。
    

## 4.2 ZMQ和CyberRT的区别

**ZMQ ：**

- ✅ 支持你在上面跑任意 payload（二进制、JSON、Protobuf……）
    
- ✅ PUB/SUB 模式下，有“topic”的概念（通过前缀或 multi-part 第一帧实现）
    
- ❌ 不管“协议 schema / IDL”，不会帮你定义结构化消息
    
- ❌ 不知道什么叫 Protobuf，它只认字节流

**CyberRT** 这类框架多做了一层

以 Apollo CyberRT 为例（ROS2 也类似）：

- 有 **Channel / Topic** 概念（字符串名）
    
- 约定了 **消息格式**：一般都用 Protobuf，`.proto` 文件直接就是接口定义
    
- 有：
    
    - 节点发现 / 通道发现
        
    - 录包 / 回放
        
    - 时间戳、时钟同步
        
    - QoS、历史缓存等
        

换句话说：

> CyberRT = ZMQ/TCP 之上 + Topic + Protobuf + 发现 + 录包等一整套“中间件生态”

## 4.3 智驾/座舱中使用 ZMQ 的思路

常见设计：

- 感知模块：PUB 多源感知结果（车道线、目标物体）。
    
- 融合模块：SUB 感知 topics，融合后再 PUB。
    
- 规划模块：SUB 融合结果，输出规划轨迹。
    
- 调试 UI：SUB 各种 debug topic，画在可视化工具上。
    

相关内容：

1. **基础模式含义**：PUB/SUB、REQ/REP 的语义。
    
2. **如何设置订阅过滤**（topic filter）。
    
3. **队列积压问题**：高频发布 + 订阅端处理不过来怎么办？
    
    - 丢旧保新、限速、背压等策略。
        
4. **多线程模型**：IO 线程 vs worker 线程的分工。

# 6 消息结构设计: Protobuf、偏移、TLV

TCP/UDP/ZMQ 解决“怎么送到”的问题，消息格式 解决“数据如何被正确、安全、长期理解”的问题。

在智驾/座舱系统中，常见的两类消息结构设计思路为：
- 高层语义信息：Protobuf
- 实时/安全信息：偏移定长或TLV(二进制协议)

## 6.1 Protobuf 

- Google 开源的**结构化数据序列化协议**。
    
- 特点：二进制、小体积、高效、跨语言。
    
- 用字段号而非内存布局描述数据结构，使用`tag + wire typr + length/value`的编码方式
    

### 6.1.1 设计理念：**兼容性优先**

在车载系统里，单车不同 ECU / 模块，以及云端服务更新频率和节奏都不同，所以 **消息格式的前向/后向兼容非常重要**。

必要原则：

1. **字段号（tag number）一旦对外发布，尽量不要改。**
    
2. 删除字段时，不要复用原字段号。
    
3. 新增字段时：
    
    - 使用新的字段号。
        
    - 对于旧版本模块，看不到这个字段，会自动忽略（自然兼容）。
        
4. 给字段起有意义的名字，但真正决定“协议版本”的是字段号而不是名字。
    

例子：在`Object` 增加一个 `acceleration` 字段：

```
message Object {
  uint32     id         = 1;
  ObjectType type       = 2;
  Point3D    position   = 3;
  double     vx         = 4;
  double     vy         = 5;
  double     heading    = 6;
  double     length     = 7;
  double     width      = 8;
  double     height     = 9;
  float      confidence  = 10;
  double     ax         = 11; // 新增：加速度 x
  double     ay         = 12; // 新增：加速度 y
}

```

旧版本不认识 11 和 12，会直接忽略，反序列化不报错；新版本则可以使用。

### 6.1.2 Protobuf 代码

编译生成：

`protoc --cpp_out=./generated perception.proto`

发送端示例：

```
perception::PerceptionFrame frame;
frame.mutable_timestamp()->set_sec(sec);
frame.mutable_timestamp()->set_nsec(nsec);
frame.set_frame_id("vehicle");

auto* obj = frame.add_objects();
obj->set_id(1);
obj->set_type(perception::VEHICLE);
obj->mutable_position()->set_x(10.0);

// 序列化为字符串 / 二进制
std::string buf;
frame.SerializeToString(&buf);

// 然后通过 TCP/UDP/ZMQ 发送 buf.data(), buf.size()
```

接收端示例：

```
perception::PerceptionFrame frame;
if (!frame.ParseFromArray(recv_buf, recv_len)) {
    // 解析失败，丢弃或记录日志
}

for (const auto& obj : frame.objects()) {
    // 正常使用结构化数据
}
```

### 6.1.3 优缺点
#### 6.1.3.1 优点
- 强语义表达能力
	- 字段有名字，有类型，有默认值
	- 嵌套结构、数组、枚举表达自然
- 兼容性计价
- 跨语言、跨平台
- 开发效率高
	- 不用再手写 `memcpy`、`struct` 对齐、大小端转换
	- 代码可读性和安全性都高
#### 6.1.3.2 缺点
Protobuf的优势，是以运行时复杂度和不确定性为代价换来的：
- 解析过程包含变长编码、分支判断、循环扫描字段
- worst-case解析时间难以严格界定
- 依赖运行时库
因此并不适合硬实时/安全关键路径

## 6.2 偏移定长消息：实时性优先

### 6.2.1 设计理念
字段在报文中的位置和长度是固定的

典型形式:`[Header][固定长度parload]`
payload内字段通过固定偏移访问
```
yaw = *(int16*)(buf + 12);
pitch = *(int16*)(buf + 14);
```
### 6.2.2 优缺点
#### 6.2.2.1 优点
- 解析时间确定
	- O(1)按偏移访问
	- worst-case可严格评估
- 无运行时依赖
- 适合R核/安全域
#### 6.2.2.2 缺点
- 扩展能力弱
- 字段变更需要新版本协议、reserved预留或多版本解析逻辑

## 6.3 TLV：在确定性和扩展性之间折中

### 6.3.1 设计理念

通过`[type] [length] [value]` 将字段定义从偏移升级为字段字典
- 认识的type->解析
- 不认识的type->按length跳过

### 6.3.2 工程优势
- 天然前向/后向兼容
- 字段可选/可增删
- 可控复杂度
	- payload长度可设上限
	- 字段数量有限

在R核侧，TLV常被视为**轻量级、可证明的ptorobuf替代方案**

# 7 ETH / CAN：底层总线的角色

## 7.1 ETH（以太网）

- 典型速率：100Mbps / 1Gbps / 2.5Gbps 等。
    
- 车载中也有 Automotive Ethernet（如 100BASE-T1）。
    
- TCP/UDP 一般跑在以太网上（标准 IP 协议栈）。
    

应用视角需要知道：

- 带宽足够传哪些数据（点云/视频流/结构化目标等）。
    
- 延迟和抖动情况（尤其是控制相关通道）。
    
- VLAN / QoS 后面可能被用来区分不同业务优先级。
    

## 7.2 CAN 总线

- 面向 ECU 间控制信号、高实时性、低带宽（典型 500kbps）。
    
- 基于 ID 仲裁，数据帧很小（一般 8 bytes，CAN-FD 可扩展）。
    
智驾/座舱应用层需要理解：

- CAN 主要承载如车速、转角、换挡位置等车辆状态信息。
    
- 数据格式一般由 OEM / Tier1 定义 DBC 文件（信号在帧中的编码方式）。
    
- 程序里更可能对接一个“CAN 抽象层/中间件”，而不是直接操作 socket。

# 8 应用层开发重点

对于智驾/座舱应用工程师，一般来说：

- **需要熟练：**
    
    - 用 socket/框架（ZMQ/ROS/Cyber）写收发逻辑。
        
    - 设计清晰的 Protobuf 消息结构。
        
    - 理解 TCP/UDP 特性并做正确选择。
        
    - 看懂简单的抓包和日志，能定位是否通信问题。
        
- **了解即可：**
    
    - TCP 拥塞控制算法细节（Cubic / BBR 等）。
        
    - 以太网物理编码（RGMII、SGMII 等）。
        
    - CAN 仲裁电平的硬件原理。
